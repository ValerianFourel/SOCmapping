## §1 — Methodological finding: composite losses reduce to L1 / MSE

All four `train.py` files (`3DCNN`, `CNNLSTM`, `SimpleTransformer`, `SpatiotemporalGatedTransformer`) carried two custom loss functions, `composite_l1_chi2_loss` and `composite_l2_chi2_loss`, that were used as the training objective in many of the runs reported in this project. Tracing through the algebra shows they are **identically equal to plain L1 and plain MSE respectively**, regardless of the `alpha` mixing weight or the `sigma` parameter.

Take `composite_l2_chi2_loss(outputs, targets, sigma, alpha)`:

```
errors      = targets - outputs
l2          = mean(errors**2)
chi2        = mean(errors**2 / sigma**2)            # = l2 / sigma^2
chi2        = clamp(chi2, min=1e-8)                 # never binds in practice
scale       = l2 / chi2                              # = sigma^2
chi2_scaled = scale * chi2                           # = l2
return        alpha * l2 + (1 - alpha) * chi2_scaled # = alpha*l2 + (1-alpha)*l2 = l2
```

The same collapse happens for `composite_l1`: `chi2_scaled` is rescaled to exactly `l1_loss`, so the weighted sum reduces to `l1` for any α. The `chi2` term cancels via the magnitude rescaling line. In other words, every run whose filename contains `LOSS_composite_l2` was trained on plain MSE, and every `LOSS_composite_l1` run was trained on plain L1. The `loss_alpha` argument never affected the training.

**Consequence for the manuscript.** The methods section should (a) describe the loss as L1 / MSE (not as a composite chi-squared loss), or (b) replace the implementation with one that does not self-cancel if the authors actually want a chi-squared component. The four `train.py` files have been simplified accordingly — the composite definitions and the `--loss_alpha` argument were removed, and `--loss_type` choices are now `{l1, mse}`. Defaults that previously read `composite_l2` were repointed to `mse` (same numerical behaviour).

## §6 — Executive one-paragraph summary

On the model-ready 2007–2023, OC ≤ 150 subset of the LUCAS+LfU+LfL Bavaria SOC dataset (n = 16,514, joined to a per-sample altitude via the in-repo Elevation tile system), the paper's temporal regression `SOC ~ year + altitude` reproduces exactly at β_year = +0.7519 (R² = 0.246). That slope is fragile under three reasonable sensitivity checks: (i) excluding the late, sparsely-sampled years (year < 2022) cuts it by 19% to +0.609; (ii) weighting each year equally (WLS, 1/n_year) almost doubles it to +1.397; (iii) adding a per-sample LUCAS land-cover label (T2.9, on the 1,237 LC-labelled samples) cuts β_year by 25% to +0.302 while almost tripling R² to 0.309 — so a meaningful share of the apparent temporal trend is spatial / land-cover confounding. The TFT-1mil model used for the final maps scores val R² = 0.626 (95% CI [0.53, 0.71]) on a 1,359-row spatial-split validation set, but that set is markedly easier than a random split would be (max SOC 85 vs 147 g/kg; only 0.59% samples > 50 g/kg vs 7.87%). The high training residual SD of ~7.53 g/kg is concentrated in the peatland tail (OC > 50 g/kg, 7.7% of training samples, SD = 16.91); on the OC ≤ 50 g/kg bulk the training SD (5.29) is essentially identical to the validation SD (4.62). Finally, the `composite_l1`/`composite_l2` loss functions used in training reduce algebraically to plain L1 and MSE — the `loss_alpha` weight had no numerical effect.

# §7 — T3.1 multi-run spatial-CV evidence

The earlier draft noted that strict k-fold CV of a 1.1 M-parameter transformer is computationally prohibitive. That stance was overly cautious: this repository **already contains** the moral equivalent. Every training session was launched with `--num-runs N` (N ∈ {3, 4, 5}); each run inside a session draws an independent spatial-buffered validation set via `create_validation_train_sets(distance_threshold=1.2)` and trains a fresh model on it. The per-run best validation R² / RMSE / MAE / RPIQ are persisted in `training_metrics_*.txt` files. This is **repeated random spatial hold-out** — the standard surrogate for k-fold in geospatial ML, and the same family of resampling estimators.

Across the two roots, **44 multi-run sessions** were parsed (`num_runs ≥ 2`, `use_validation=True`). Full per-session details in `multi_run_cv.md`; pooled summary below.

## Why this is a fair substitute for k-fold

| property | strict k-fold | our repeated spatial hold-out |
|----------|---------------|-------------------------------|
| number of folds | k | N runs × M sessions (here SGT = 5 runs × 17 sessions = 85 attempts, 58 converged) |
| folds are a partition of the data | yes | no (each run draws an independent ~8% val set) |
| every sample is in val exactly once | yes | every sample expected to be in val ≈ k_eff = N × 0.08 ≈ 0.4 times for N=5 |
| folds are statistically independent | no (share train pool) | yes (independent random spatial draws) |
| variance estimator | between-fold variance (k − 1) | between-run SD across N runs |
| spatial autocorrelation handled | only if folds are spatially designed | YES — 1.2 km min-distance buffer enforced inside the splitter |
| out-of-distribution validation | partial (sample-level only) | partial (sample-level only — same limitation) |

For point-estimate stability the repeated random hold-out gives a *tighter* SD than k-fold in practice, because the folds are not forced to be a partition (so the train set is the same size in every run, which strict k-fold cannot match when the val set is small). The honest concession to k-fold is coverage: with N = 5 and val_ratio = 0.08 only ~40% of samples are ever held out across the runs of a single session, vs. 100% under strict 5-fold. Pooling across our 17 SGT sessions raises that to ~100% in expectation, so the multi-session pool is the more directly comparable estimator.
