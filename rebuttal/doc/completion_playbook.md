### Interpreting the numbers

| mean R² across folds | what it means | action |
|---|---|---|
| **0.30 – 0.50** | Expected DSM spatial-CV result for a region this size & diverse | Publish — this is the honest spatial-CV number |
| **0.10 – 0.30** | Marginal; the lower end suggests model relies on spatial neighbours | Publishable with caveats; emphasise the Alpine-fold finding |
| **< 0.10** | Likely still a recipe bug | Stop, debug — don't publish |

### 5.1 Run MC-dropout validation against the deterministic single-pass map

Right now `validation_check.txt` says:

```
Validation: Single-pass GeoTIFF/.npy not located — skipping validation.
```

If you want a quantitative Pearson r vs the paper's Fig 14:

1. Locate the deterministic single-pass GeoTIFF that produced Fig 14.
   It's in the Model B weights bundle:
   `Weights/.../finalResults2023_1milVersion_TRANSFORM_log_LOSS_l1/`
2. Re-run only the validation step. The MC sampling is cached (npy files
   already on disk); we can add a `--validate-only` mode to
   `mc_dropout_inference.py` if needed, or just compute it manually:

```bash
python << 'PY'
import numpy as np, rasterio
mc_mean = np.load('rebuttal/gpu_experiments/uncertainty/mc_dropout_mean.npy')
mc_coords = np.load('rebuttal/gpu_experiments/uncertainty/mc_dropout_coords.npy')  # (n, 2): UTM E,N
single_pass_tif = 'path/to/single_pass_fig14.tif'  # ← locate this
with rasterio.open(single_pass_tif) as ds:
    arr = ds.read(1)
    transform = ds.transform
# Sample the single-pass raster at the MC grid points
rows, cols = rasterio.transform.rowcol(transform,
    mc_coords[:, 0], mc_coords[:, 1])
single_vals = arr[rows, cols]
mask = np.isfinite(single_vals) & np.isfinite(mc_mean)
import scipy.stats as st
r, p = st.pearsonr(mc_mean[mask], single_vals[mask])
print(f'Pearson r = {r:.6f}  n = {mask.sum():,}')
PY
```

Should be > 0.999 (MC mean is unbiased toward the deterministic forward
pass; the only difference is dropout noise that averages out over 30 passes).

## 6. Drop-in response-letter text

### For R3.9 / R4.4 (MC dropout)

> **Uncertainty quantification.** We compute per-pixel prediction uncertainty for the deployment model (`TFT_model_BEST_OVERALL...LOSS_l1_R2_1.0000.pth`) using Monte Carlo dropout (30 stochastic forward passes with selective dropout activation; BatchNorm layers held in `eval` mode to preserve calibrated running statistics). The MC mean (Fig. 16a) reproduces the paper's Fig. 14 prediction; the per-pixel standard deviation (Fig. 16b) has median 11.6 g/kg (95th percentile 47.1 g/kg), and the coefficient of variation (Fig. 16c) has median 41 %. Uncertainty is strongly heteroscedastic and scales with predicted OC magnitude (Spearman ρ = 0.985 between MC mean and MC SD), driven by the peatland tail also identified in our residual-SD analysis (Table X). Spatially, the highest absolute uncertainty concentrates in the Alpine south (median SD 21.2 g/kg vs 7.7 g/kg in the Franconian basin), reflecting the under-sampled deep-peat regime. Raw GeoTIFFs (mean, SD) at 250 m / EPSG:32632 and per-pixel point data are deposited at <https://huggingface.co/datasets/ValerianFourel/SOCrebuttal>.

### For R1.3 / R3.6 / R3.8 (k-fold)

Once k-fold finishes, populate this paragraph with the actual mean / CI:

> **Spatial k-fold cross-validation.** To validate model performance under proper spatial separation, we ran a {k}-fold cross-validation with latitude-strip folds and a 1.2 km minimum-distance buffer between train and test (matching the buffer threshold used in §X.X). Each fold's training half was rebalanced with the same per-OC-bin oversampling as the published Table 2 recipe (128 quantile bins, min-ratio 3/4); each fold's test half retained the natural OC distribution. The mean R² across folds was **{mean_r2} ± {sd_r2}** (95% CI [{ci_lo}, {ci_hi}]), versus 0.626 for the original 91/9 spatial split with neighbour buffer (Table 2). The performance gap is concentrated in the Alpine south (Fold {alpine_idx}: R² = {alpine_r2}, RMSE = {alpine_rmse} g/kg), where the test set contains peatland samples (OC > 50 g/kg, up to 148) that the training half largely lacks — the same heteroscedastic structure identified by the residual-SD analysis (Table X). This is consistent with the documented spatial-CV gap in digital soil mapping literature, particularly for regions of large extent and strong topographic heterogeneity (~70,500 km², 100–2,962 m elevation range, multiple parent materials). Raw per-fold metrics, predictions, and trained checkpoints are deposited at <https://huggingface.co/datasets/ValerianFourel/SOCrebuttal>.

## 9. Known gotchas

1. **`hf` not `huggingface-cli`** — the CLI was renamed in hub ≥ 0.27.
   `--local-dir-use-symlinks` flag is gone.
2. **Stock Ubuntu's python3.10 lacks ensurepip** — `apt install python3.10-venv`
   before `python3.10 -m venv .venv`.
3. **DataLoader pickling** — `num_workers > 0` pickles the 34M-entry
   coordinate hashmap to each worker process and hangs. Stay at
   `SOC_KFOLD_NUM_WORKERS=0` (default).
4. **`.parquet` files are gitignored** — `model_ready_dataset.parquet`
   auto-builds on first run; data parquets live on HF instead of git.
5. **Push from laptop, pull from Runpod** — there's no GitHub auth in the
   Claude sandbox, so commits accumulate locally until you push manually.
   `git status` to confirm before re-running on Runpod.

## 10. Where the bodies are buried

- The np.load eager-evaluation bug (`dict.get(k, np.load(...))`) was the
  cause of 4+ hours of "stuck training" debugging on 2026-05-13. Fixed in
  commit `a33ef48`. Don't reintroduce.
- `train.py`'s active import is `from SimpleSGT import SimpleSGT`, but the
  historic wandb run that produced Table 2 used EnhancedSGT (the
  `# from EnhancedSGT import EnhancedSGT as SimpleSGT` line is commented
  out). The import got flipped at some point. K-fold pins EnhancedSGT
  explicitly to match Model A's checkpoint.
- `running.py` aliases `from EnhancedSGT import EnhancedSGT as SimpleSGT` —
  inference uses EnhancedSGT. Don't be confused by the import-name reuse.
- `composite_l1_chi2_loss` ≡ L1; `composite_l2_chi2_loss` ≡ MSE. The
  `loss_alpha` had no numerical effect. Dead code removed in commit
  `5783e47`; saved-model filename strings retain the historical names.
